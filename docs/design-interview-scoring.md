# 設計面談（Design Interview）採点・出題ガイド

## 概要

Naitei の設計面談コースは、候補者の設計スキル・業務知識・コミュニケーション能力を総合的に評価するシステムです。

**バージョン:** 2.0（2026年4月）  
**更新内容:** 出題数を 10 題 → 14 題に拡大、平均分制を導入し100点制を維持

---

## 出題方式

### 構成

総 **14 題**（従来 10 題 → 4 題追加）

```
固定必考（9題）:
├─ Q1-Q3    基本設計の範囲（3題）
├─ Q7-Q10   基本設計の領域（4題）
├─ Q11      基本設計のプロセス（1題）
└─ Q28      AI ツール利用（1題）

動的選択（5題）:
├─ Q4-Q6        基本設計の経験（3題候補）
├─ Q12-Q21      通常問題（10題候補）
├─ Q22-Q26      加点減点問題（5題候補）⭐NEW
├─ Q27          補足事項（1題候補）⭐NEW
└─ 計 19 題から選択、設計領域（domain）関連を優先
```

### 題目分類表

| 範囲 | 題号 | 分類 | 内容 | 難度 | 出現率 |
|------|------|------|------|------|--------|
| 固定 | Q1-Q3 | 基本範囲 | AO/SI経験、基本設計参画、リード経験 | 必須 | 100% |
| 固定 | Q7-Q10 | 設計領域 | 3つの設計領域の詳細説明 | 必須 | 100% |
| 固定 | Q11 | プロセス | 基本設計の作業・成果物 | 必須 | 100% |
| 固定 | Q28 | 補足 | AI ツール利用 | 必須 | 100% |
| **動的** | **Q4-Q6** | 経験 | 顧客対応、設計領域、大規模案件 | 通常 | **~66%** |
| **動的** | **Q12-Q21** | 通常 | 設計プロセス、DB設計、I/F設計など | 通常 | **~50%** |
| **動的** | **Q22-Q26** | 加減 | テスト方針、Client Facing など | 加点対象 | **~40%** |
| **動的** | **Q27** | 補足 | 心得・注意点 | 加点対象 | **~40%** |

## 採点方式

### 背景評価（Background Assessment）

**配点:** 0 ～ 50 点

各項目ごとのスコアを合計：

| 項目 | B級 | C級 | 3年未満 | 1～3年 | 3年以上 |
|------|-----|-----|--------|--------|---------|
| 日本語レベル | 0pt | 10pt | - | - | - |
| ソフトスキル | 5pt (S1) | 10pt (S2) | 15pt (S3) | - | - |
| 基本設計年数 | - | 0pt | 5pt | 10pt | - |
| 要件定義経験 | - | 0pt | - | 10pt | - |
| レビュー経験 | - | 0pt | - | 5pt | - |
| **最大** | - | - | - | - | **50pt** |

### 技術スコア（Technical Score）

**配点:** 0 ～ 50 点

#### 計算方式（平均分制 / Average-based Normalization）

```
技術スコア = (全設問の得点合計 ÷ 設問数) × 10
```

**特徴:**
- 設問数が変わっても、スコア基準は一定
- 各設問の重みが等しい
- 最大値は常に 50 点

**計算例:**

| 設問数 | 各問平均 | 計算式 | 技術スコア |
|--------|---------|--------|-----------|
| 10題 | 4.0/5 | 4.0 × 10 | 40点 |
| 14題 | 4.0/5 | 4.0 × 10 | 40点 ✅ |
| 14題 | 5.0/5 | 5.0 × 10 | 50点 |
| 14題 | 3.0/5 | 3.0 × 10 | 30点 |

#### 各設問の採点（0～5点）

| スコア | 正確性 | 網羅性 | 明瞭性 | 専門用語 |
|--------|--------|--------|--------|----------|
| **5点** | 完全に正確、実務的 | すべての観点網羅 | 論理的で構造化 | 適切かつ豊富 |
| **4点** | 概ね正確、細部に不足 | 主要事項押さえている | 概ね明瞭 | 主要用語を使用 |
| **3点** | 方向性は正しい | 半分以上カバー | 伝わるが飛躍あり | いくつかの用語 |
| **2点** | 誤解や混乱 | 大きな漏れあり | 説明が断片的 | 限定的、誤用あり |
| **1点** | 基本的な理解欠陥 | ほとんど欠けている | 意図が伝わりにくい | ほぼなし |
| **0点** | 全く正確でない / 無回答 | 実質的な内容がない | 意味をなさない | なし |

### 総合スコア（Total Score）

**配点:** 0 ～ 100 点

```
総合スコア = 背景評価スコア + 技術スコア
           = (0～50) + (0～50)
           = 0～100点
```

## P レベル判定（Proficiency Level）

### 基準

| P級 | スコア | 説明 | 対応ML | 備考 |
|-----|--------|------|--------|------|
| **P4** | ≥90 | 要件担当者向け / システム要件定義参画可 | ML7/ML8 | 設計のみならず要件定義も理解 |
| **P3** | ≥80 | 設計高級者 / 複雑機能担当・Review 可 | ML8/ML9 | 前後設計経験、レビュー能力有 |
| **P2** | ≥60 | 設計中級者 / Medium 機能担当可 | ML9/ML10 | Backend/Front 設計経験あり |
| **P1** | <60 | 設計初心者 / Simple 設計のみ、SV指導必要 | - | トレーニング・P1試験が必要 |

### 判定例

| 背景分 | 技術分 | 合計 | 各問平均 | P級 |
|--------|--------|------|---------|------|
| 30 | 40 | 70 | 4.0/5 | P2 |
| 40 | 45 | 85 | 4.5/5 | P3 |
| 50 | 50 | 100 | 5.0/5 | P4 |
| 20 | 35 | 55 | 3.5/5 | P1 |

## 設計領域（Design Domains）

出題時に、候補者が選択した設計領域に関連する問題を優先的に選択します。

### 選択可能な設計領域

1. **画面設計** — UI/UX、画面遷移、入力検証など
2. **バッチ設計** — バッチジョブ、スケジュール、エラー処理など
3. **帳票設計** — レポート、出力形式、配置設計など
4. **セキュリティ設計** — 認証、暗号化、権限管理など
5. **データベース設計** — テーブル設計、ER図、正規化など
6. **API設計** — RESTful API、I/F仕様、エラーハンドリングなど
7. **外部システムI/F設計** — システム連携、データ形式など
8. **要件定義** — ビジネス要件、スコープ管理など

### Domain 関連付けの仕組み

```
候補者が「画面設計」「API設計」を選択
  ↓
Q5（画面/バッチ/帳票）、Q16（DB設計）、Q18（I/F設計）
などが優先的に選出される
  ↓
より実務的で候補者の経験に即した評価が可能
```

## よくある質問

### Q: 10題のときと14題のときで、同じ得点でも P レベルが違うのでは？

**A:** いいえ、平均分制により同じです。

例）各問平均 4.0/5 の場合、常に技術分 = 40点です。
- 10題で 4.0/5 平均 → 技術分 40点
- 14題で 4.0/5 平均 → 技術分 40点 ✅

### Q: 加点減点問題（Q22-Q26）は減点になることもある？

**A:** いいえ。「加点減点問題」は出題構成の区分であり、スコアは他の問題と同じ 0～5点です。

- 加点のチャンスは、「得点が高い」こと
- 減点はありません

### Q: Q27は なぜ「補足」なのに出題される？

**A:** AI ツール利用（Q28）は必須、心得・注意点（Q27）は加点対象として位置付けられています。

出題確率 ~40%で、候補者の深い専門性・リーダーシップを評価できます。

### Q: 域（Domain）関連でない問題がランダムに選ばれるのか？

**A:** はい。選択ドメイン関連の問題を優先しますが、5題すべてが関連問題で埋まるわけではありません。

```
優先順位:
1. Domain 関連の通常問題 / 加点減点問題
2. Domain 非関連の通常問題 / 加点減点問題
3. （都度的に有効な問題がない場合はスキップ）
```
